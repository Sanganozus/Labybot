1 /**
2  * @file pathFollower.h
3  * @ingroup pathFollower
4  *
5  * Implementation of a path follower which interfaces with HWPCS.
6  * Additionally, the path follower can be controlled directly by functions
7  * such as pathFollower_setNewPath() and pathFollower_command().
8  *
9  * The path follower internally manages two states of type PathFollowerStatus_t
10  * and PathFollowerExtStatus_t. These states are updated by calls to
11  * pathFollower_update(). The current state can be acquired with
12  * pathFollower_getStatus() and pathFollower_getExtStatus().
13  *
14  * In order to have HWPCS reflect the recent state and update the Scene view accordingly,
15  * the state returned by pathFollower_getStatus() must be sent regularly
16  * to HWPCS.
17  *
18  * You are welcome to modify the path follower according to your needs.
19  *
20  * When compiling with the symbol PATHFOLLOWER_QUIET defined, no informational
21  * messages are sent to HWPCS. This includes information when a path has been
22  * set, following is started, paused or reset, path following parameters are
23  * set, end of path is reached and a switch to another segment occurs. Only
24  * error messages (such as when a path can not be set or when the path follower
25  * cannot find the orthogonal projection or lookahead point) are transmitted to
26  * HWPCS. PATHFOLLOWER_QUIET saves 428 bytes of flash memory and 32 bytes of
27  * SRAM.
28  *
29  * The error messages can be suppressed when additionally compiling with the
30  * symbol PATHFOLLOWER_REALLY_QUIET defined (only applicable if
31  * PATHFOLLOWER_QUIET is defined as well).
32  * PATHFOLLOWER_REALLY_QUIET additionally saves 278 bytes of flash memory.
33  */

127 /**
128  * Calculate and determine all the fields in the current PathFollowerStatus_t
129  * and PathFollowerExtStatus_t (which are internally managed and can be acquired
130  * with pathFollower_getStatus() and pathFollower_getExtStatus())
131  * from the current robot position and the path (either received by the
132  * communication protocol or manually set by pathFollower_setNewPath()) in case
133  * path following is enabled.
134  *
135  * This function should be called from <code>main()</code> context in regular intervals.
136  *
137  * The function operates in two steps:
138  * 1. First, the location of the orthogonal projection of the current pose onto the path
139  * is determined. The function determines, starting from the current segment (in
140  * PathFollowerStatus_t.segStart and PathFollowerStatus_t.segEnd) along subsequent
141  * path segments, which orthogonal projection of the current position onto the
142  * segments has smallest distance to the current position. In order to keep
143  * execution time acceptable and also to avoid taking shortcuts (or skipping
144  * too many segments) in case of concave or crossing paths, the number of
145  * consecutive segments examined is limited by the value supplied to
146  * pathFollower_setSegmentLimit().
147  * 2. Once the location of the orthogonal projection is found in step 1, the
148  * location of the lookahead point is determined by advancing the distance
149  * given by pathFollower_setLookaheadDistance() along the path, even over
150  * multiple segments if necessary.
151  *
152  * If the function returns true, a controller should generate control inputs for
153  * approaching the lookahead point.
154  * If the function returns false, the robot should stop and no further control
155  * inputs should be applied.
156  *
157  * Use timeTask_getTimestamp() and timeTask_getDuration() timeTask_getDuration()
158  * to measure the execution time of this function as it depends heavily on the
159  * values given by pathFollower_setSegmentLimit() and
160  * pathFollower_setLookaheadDistance(). Note that the first run of
161  * pathFollower_update() after a new path has been set takes much longer than
162  * following runs since several internal variables need to be initialized.
163  *
164  * @param currentPose current pose of the robot
165  * @return true if the internally managed state structures were updated and
166  * following is still enabled,
167  * false if following was disabled in case of reaching the path's end
168  */
169 bool pathFollower_update(const Pose_t* currentPose);

222  *
223  * Fields PathFollowerStatus_t.segStart and PathFollowerStatus_t.segEnd denote the
224  * path segment on which the orthogonal projection of the robot's pose onto the path was
225  * found.
226  *

231  * generating control commands for the robot.

234  * time after pathFollower_update() completed in order to have HWPCS evaluate it such
235  * that all information is displayed in the Scene View. This also allows HWPCS to detect
236  * the status of the path follower, e.g. when the robot reached the end of the path.
237  *
238  * @return a pointer to the structure of type PathFollowerStatus_t which is updated by
239  * pathFollower_update().

270 /**
271  * Set the distance in mm between orthogonal projection of current position onto path
272  * and lookahead point. The distance must be positive.
273  * Using pathFollower_update(), the lookahead point (computed in an internally managed
274  * structure of type PathFollowerStatus_t, which can be fetched with
275  * pathFollower_getStatus()) is always kept on the path ahead of the orthogonal
276  * projection by this distance.
277  *
278  * Note that when updating this value while the path follower is running, the new
279  * lookahead distance must be less than ten times the old one, otherwise the
280  * lookahead point cannot be located. This is because the extrapolation of an
281  * additional point at the end of the path (at distance of ten times the
282  * lookahead distance after the last point) is only done when a new path is set.
283  * An updated value of the lookahead distance is only taken over by the
284  * path follower when a switch of the orthogonal projection onto the path to
285  * another segment occurs.
286  *
287  * Default is 1mm.
288  *
289  * @param lookaheadDistance the lookahead distance given in millimeters
290  */
291 void pathFollower_setLookaheadDistance(const float lookaheadDistance);

304 /**
305  * Set the number of segments (starting from the current one) to look ahead when
306  * locating the orthogonal projection of the current position onto the path.
307  * The minimum value is 1, indicating that only the current and next segment
308  * are taken into account for locating the lookahead point. A value of 0 is
309  * ignored.
310  *
311  * This value should not be set too large, as this dramatically increases the
312  * execution time of pathFollower_update().
313  * Use timeTask_getTimestamp() and timeTask_getDuration() to measure the execution
314  * time in order to select a feasible value for the segment limit.
315  *
316  * Additionally, a large value can cause the path following algorithm to take
317  * unwanted shortcuts in case of crossing or concave paths.
318  *
319  * It is however necessary, to deal with the problem of a concave path or a path
320  * which crosses itself.
321  *
322  * Default is 5.
323  *
324  * @param segmentLimit the segment limit which is used by pathFollower_update()
325  */
326 void pathFollower_setSegmentLimit(const uint8_t segmentLimit);
